# 부동산 경매 교육 멀티에이전트 RAG (가칭)

부동산 법원경매를 **가르치고 연습시키는** 학습용 AI 서비스.
오케스트레이터가 질문을 전문 에이전트로 분배하는 멀티에이전트 RAG 구조.

> ⚠️ 본 서비스는 **교육·학습용**입니다. 특정 물건에 대한 투자·법률 자문이 아니며,
> 실제 입찰 전에는 반드시 전문가 확인이 필요합니다. 모든 답변에 출처(조문)를 함께 제시합니다.

## 프로젝트 소개

경매 절차는 현행 기준(민사집행법, 2026년 5월 기준)으로 확인했습니다. 절차 자체는 안정적이고, 교육 프로젝트의 진짜 핵심은 "절차 암기"가 아니라 **권리분석**이라는 점이 설계의 출발점이 됩니다. 먼저 절차를 흐름도로 정리하고, 그 위에 교육 설계와 역할 분담을 얹겠습니다.

[경매 절차 흐름도]

위 흐름이 법원경매의 기본 절차입니다. 그런데 교육 프로젝트를 설계할 때 가장 중요한 인사이트는 이겁니다 — **절차(흐름) 자체는 외우면 끝나는 쉬운 부분이고, 학습자가 실제로 돈을 잃고 어려워하는 지점은 "권리분석"** 이라는 거예요. 여기에 AI 교육의 가치가 집중되어야 합니다.

## 핵심 개념

### 권리분석 (Rights Analysis)

권리분석의 출발점은 **말소기준권리**입니다. 등기부에서 (근)저당권·(가)압류·담보가등기·경매개시결정 등 중 가장 앞선 것을 기준선으로 잡고, 그보다 앞선 권리는 매수인에게 **인수**, 뒤선 권리는 **말소**된다는 원리죠. 등기부에서 말소기준권리를 빠르게 찾아낸 뒤 그보다 앞선 날짜의 권리가 있는지 확인하는 것이 권리분석의 본질이며, 이 기준선 하나만 정확히 이해해도 사고의 대부분을 예방할 수 있습니다.

가장 까다로운 쟁점은 **임차인의 대항력**입니다. 임차인이 전입신고·점유 등 대항요건을 말소기준권리 설정일보다 앞서 갖추면 대항력이 있고, 같은 날이거나 이후면 대항력이 없습니다. 대항력 있는 임차인의 보증금은 낙찰자가 떠안을 수 있어서, 여기서 초보자가 가장 크게 손해를 봅니다. 또 유치권·법정지상권·분묘기지권 등은 등기 순위와 무관하게 인수되는 함정도 있죠. 즉 교육 콘텐츠의 70~80%는 절차가 아니라 이 권리분석 영역에 들어가야 합니다.

## 설계 원칙 및 고려사항

경매는 "실제 돈과 법적 책임"이 걸린 도메인이라, 일반 챗봇과 다른 설계 원칙이 필요합니다.

첫째, **포지셔닝을 "학습/교육용"\*\***으로\***\* 명확히 못 박을 것.** 특정 물건에 대해 "이거 사도 됩니다 / 권리분석상 안전합니다"라고 단정하면 곧바로 투자자문·법률자문 책임 문제가 생깁니다. AI는 개념을 가르치고 분석을 _연습시키는_ 역할까지만. 결론 화면마다 "참고용이며 실제 입찰 전 전문가 확인 필요" 디스클레이머를 둡니다. 이건 발표에서도 "우리는 리스크를 인지하고 설계에 반영했다"는 강점으로 어필됩니다.

둘째, **데이터 출처는 공공·법령 중심으로.** 데모 단계에서 상업 경매정보(지지옥션 등) 크롤링은 비용·약관 리스크가 있으니 피하고, 무료·공식 소스로 지식베이스를 구성하세요.

| **용도**       | **소스**                                                       |
| -------------- | -------------------------------------------------------------- |
| 법령(RAG 핵심) | 국가법령정보센터 — 민사집행법, 주택·상가임대차보호법           |
| 절차·해설      | 찾기쉬운 생활법령정보(easylaw), 대법원 법원경매정보            |
| 실물 자료      | 매각물건명세서·현황조사서·감정평가서 샘플(법원경매정보 공개분) |
| 연습용         | 직접 제작한 가상 매각물건명세서 + 정답 권리분석                |

셋째, **RAG가 잘 맞는 곳과 위험한 곳을 분리.** 절차 안내·용어 설명·법령 인용·개념 학습은 RAG가 강점을 발휘합니다. 반면 특정 물건의 정확한 권리분석 단정이나 배당금액 계산은 환각(hallucination) 시 실제 피해로 직결되므로, "가상 사례 기반 학습/연습"으로 한정하고 항상 **출처 조문을 인용**하게 하세요.

## 시스템 아키텍처

### 멀티에이전트 RAG 구조 (Multi-Agent RAG Structure)

프로그램 주제(에이전트)와 보유 스택(Streamlit + GPT-4o-mini + ChromaDB + LangChain + 한국어 임베딩)에 그대로 얹어, **오케스트레이터가\*\*** 질문을 4개 전문 에이전트로 라우팅\*\*하는 구조를 권합니다. 멀티에이전트라 데모 임팩트도 큽니다.

이전에 작업하신 **법률 문서 clause-aware \*\***청킹\*\*이 그대로 핵심 자산이 됩니다 — 법령을 조문 단위로 끊어 ChromaDB에 넣으면, 답변마다 정확한 조문 인용이 가능해져 신뢰성 문제를 상당히 해결합니다.

**에이전트 구성 (목표)**

- **오케스트레이터(Router)**: 질문 의도를 분류해 적절한 에이전트로 분배 (LangChain 라우터가 질문 의도를 분류해 분배)
- **① 절차 안내**: 경매 흐름·단계 Q&A ("지금 단계 다음은?", "배당요구 종기란?" 등 흐름 Q&A)
- **② 권리분석 튜터**: 말소기준권리·대항력을 가상 사례로 단계별 학습 (단정 금지, 학습 유도형)
- **③ 법령·용어 사전**: 조문/용어를 출처와 함께 인용
- **④ 사례·퀴즈**: 가상 매각물건명세서로 권리분석 문제 출제·채점
- **⑤ AI 입찰가 예측 에이전트 (Price Oracle)**: 목표 수익률 역산 기반 최대 입찰가 마지노선 제안, 법원별 낙찰가율 통계 기반 낙찰 확률 시뮬레이션

ChromaDB는 컬렉션을 법령 / 절차해설 / 용어집 / 가상사례로 나눠 라우팅 정확도를 높이면 됩니다.

### 지식베이스(ChromaDB) 4개 컬렉션

| 컬렉션               | 내용      | 데이터                         |
| -------------------- | --------- | ------------------------------ |
| `auction_laws`       | 법령 조문 | `data/laws/*.md`               |
| `auction_procedures` | 절차 해설 | `data/procedures/*.md`         |
| `auction_glossary`   | 용어집    | `data/glossary/glossary.json`  |
| `auction_cases`      | 가상 사례 | `data/cases/sample_cases.json` |

## 실행 준비

```bash
# 1) 가상환경
python -m venv venv && source venv/bin/activate   # Windows: venv\Scripts\activate

# 2) 설치
pip install -r requirements.txt

# 3) 키 설정
cp .env.example .env   # 그리고 .env 안의 키를 채움
```

> 실제 동작(ingest/app)은 2단계 이후 코드가 추가되면 진행합니다.

### 동작 테스트

```bash
python ingest.py            # chroma_db/ 구축 (최초 1회, 키 필요)
python cli.py "배당요구 종기가 뭐야?"   # 절차 안내 에이전트에 질문
```

### 앱 실행

```bash
python ingest.py            # 최초 1회 DB 구축
streamlit run app.py        # 웹 UI 실행 (채팅 + 퀴즈 채점)
```

## AI 입찰가 예측 기능 (Price Oracle)

> **에이전트명**: Price Oracle — 수익성은 극대화하고 오버페이(단독 낙찰)는 방지하는 ML 기반 입찰 시뮬레이터

### 기능 목적

경매 초보자와 전업 투자자 모두가 가장 어려워하는 "얼마에 써야 낙찰되는가?"에 과학적 근거를 제시합니다.
해당 법원의 최근 3년 낙찰가율 데이터, 물건 종류별 경쟁률, 현 시점 주택담보대출 금리, KB 시세 등을 종합 분석하여 입찰 성공 확률 분포를 제안합니다.

### 입력 데이터

| 입력 항목      | 설명                                          |
| -------------- | --------------------------------------------- |
| 물건 유형      | 아파트 / 다세대·빌라 / 오피스텔 / 상가 / 토지 |
| 감정가         | 법원 감정 기준 금액 (원)                      |
| 현재 최저가    | 유찰 횟수 반영 최저 입찰가 (원)               |
| 소재 법원      | 관할 경매 법원 (예: 서울중앙지법)             |
| 목표 세후 수익 | 사용자가 원하는 최종 순수익 (원)              |
| 예상 대출 비율 | LTV 기준 대출 비중 (%)                        |
| 적용 금리      | 주택담보대출 연이율 (%)                       |

### 출력 결과

| 출력 항목              | 설명                                                   |
| ---------------------- | ------------------------------------------------------ |
| **최대 입찰 마지노선** | 목표 수익 달성을 위한 최대 입찰 허용 금액              |
| **비용 역산 내역**     | 취등록세·법무비·명도비·인테리어비·대출이자 항목별 상세 |
| **낙찰 확률 분포**     | 입찰가 구간별 낙찰 확률 분포 차트                      |
| **예상 경쟁자 수**     | 해당 법원·물건 유형 기반 예상 입찰 인원 범위           |
| **AI 코멘트**          | 현재 시장 상황과 물건 특성을 반영한 입찰 전략 요약     |

### 기술 구현 방향

- **Gemini API + Google Search Grounding**: 실시간 해당 법원 낙찰가율 및 시세 조회
- **ChromaDB `auction_cases` 컬렉션**: 과거 유사 낙찰 사례 벡터 검색으로 RAG 컨텍스트 보강
- **Streamlit UI**: `st.number_input`, `st.slider`, `st.selectbox`로 파라미터 입력 → `st.bar_chart` / `st.metric`으로 결과 시각화
- **면책 고지**: 모든 예측 결과는 교육·참고용이며 실제 입찰 전 전문가 확인 필요

## 팀 역할

프로젝트의 성공적인 진행을 위해 다음과 같은 역할 분담을 제안합니다. 특히 데모 시점(6/26 기준)이 임박했으므로 병렬적인 작업이 필요합니다.

- **팀장/기획·도메인**: 전체 시나리오·UX 설계, 권리분석 교육 콘텐츠 설계(가상 사례·정답 세트 제작), 오케스트레이터 라우팅 규칙 정의, 발표·데모 시나리오. 도메인을 가장 잘 아는 사람이 콘텐츠 정합성을 책임집니다.
- **데이터·지식베이스**: 법령/절차/용어 수집·정제, clause-aware 청킹, 한국어 임베딩, ChromaDB 4개 컬렉션 구축.

- **프론트·통합**: Streamlit UI(에이전트별 탭/챗), 통합, Streamlit Cloud 배포, 사전 QA.
